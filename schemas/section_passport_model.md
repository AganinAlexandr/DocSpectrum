# Логическая модель паспорта раздела

## 1. Назначение документа

Этот документ описывает логическую модель `паспорта раздела` для проекта `DocSpectrum`.

Цель документа - зафиксировать структуру данных на уровне смысловых блоков, сущностей, связей и правил, не переходя пока к формальной JSON Schema.

Паспорт раздела должен стать основной единицей хранения, повторного анализа и сравнения разделов проектной документации.

В архитектурном смысле `section_passport` следует рассматривать как доменный профиль более общего понятия `document_passport`. Ядро модели должно описывать документ и его элементы, а профиль проектной документации добавляет поля раздела, подраздела, объекта, стадии и отраслевых классификаторов.

## 2. Назначение паспорта раздела

Паспорт раздела нужен для того, чтобы один раз извлечь и нормализовать данные из раздела, а затем многократно использовать их для:

- сравнения с библиотекой разделов;
- поиска ближайших аналогов;
- оценки шаблонности и самостоятельности;
- сравнения версий;
- построения векторных признаков;
- визуализации проекций пространства признаков;
- последующей корпусной аналитики.

Паспорт должен хранить не только результат распознавания, но и качество, происхождение и ограничения полученных данных.

Важно:

паспорт раздела не должен хранить результаты сравнения с библиотекой. Он должен описывать только конкретный раздел и его собственные признаки. Все выводы, зависящие от текущего состава библиотеки, должны храниться в модели `результата сравнения`.

## 3. Принципы модели

### 3.1. Паспорт не равен исходному PDF

Паспорт не должен быть копией PDF и не обязан хранить все сырые данные.

Он хранит:

- идентификацию раздела;
- нормализованные сущности;
- признаки;
- сигнатуры;
- связи;
- показатели качества;
- данные, необходимые для повторного сравнения.

### 3.2. Паспорт должен быть расширяемым

Модель должна выдерживать появление новых классов элементов, новых осей анализа и новых признаков без разрушения уже сохранённых паспортов.

Модель также должна выдерживать появление новых доменных профилей. Для проектной документации базовой единицей остаётся `раздел`, но для других видов документации аналогичный паспорт может описывать отдельный документ или документ внутри комплекта.

Для этого в модели должны быть:

- версия схемы;
- стабильные идентификаторы сущностей;
- явное разделение обязательных и расширяемых блоков;
- место для дополнительных признаков и метаданных.

### 3.3. Паспорт должен различать отсутствие и неизвестность

Модель должна различать:

- признак измерен и равен нулю;
- признак неприменим к данному разделу;
- признак не был извлечён;
- признак извлечён с низкой уверенностью.

Это критично для векторного сравнения и расчёта информативного подпространства.

### 3.4. Паспорт должен поддерживать многоосевую модель

Паспорт должен хранить данные для основных осей анализа:

- структурной;
- текстовой;
- графической;
- оси связей текста и графики;
- оси типовых групп элементов.

### 3.5. Паспорт должен быть независим от библиотеки

Паспорт раздела должен оставаться стабильным при добавлении новых разделов в библиотеку.

Это означает:

- паспорт хранит признаки конкретного раздела;
- паспорт не хранит библиотечные совпадения;
- паспорт не хранит ранги похожих разделов;
- паспорт не хранит выводы о заимствованиях;
- паспорт не должен пересчитываться только потому, что в библиотеку добавлен новый раздел.

Сравнение с библиотекой выполняется отдельной моделью результата сравнения, которая берёт признаки из паспорта и сопоставляет их с библиотечными данными.

## 4. Верхнеуровневая структура паспорта

Рекомендуемая логическая структура:

```text
section_passport
  schema
  identity
  source
  extraction
  document_structure
  elements
  typical_element_groups
  text_axis
  graphics_axis
  text_graphics_links
  feature_vectors
  quality
  lineage
  extensions
```

## 5. Блок `schema`

Блок описывает версию и совместимость модели паспорта.

Рекомендуемые поля:

- `schema_name`
- `schema_version`
- `schema_status`
- `created_by`
- `created_at`
- `compatible_with`

Назначение:

- понять, по какой версии правил сформирован паспорт;
- обеспечить миграции;
- различать черновые и стабильные версии.

## 6. Блок `identity`

Блок идентифицирует раздел как предмет анализа.

Рекомендуемые поля:

- `section_id`
- `object_id`
- `project_id`
- `organization_id`
- `designer_id`
- `section_type`
- `subsection_type`
- `stage`
- `version_label`
- `document_date`
- `crc32`

Примечания:

- `designer_id` может быть неизвестен или заполняться позднее.
- `crc32` фиксирует состояние содержимого раздела и помогает отличать версии.
- `project_id` нужен для будущего анализа генома проекта целиком.

## 7. Блок `source`

Блок описывает исходные файлы и происхождение данных.

Рекомендуемые поля:

- `source_file`
- `source_file_name`
- `source_file_path`
- `source_file_size`
- `source_file_crc32`
- `source_format`
- `source_origin`
- `source_created_at`
- `source_modified_at`
- `related_files`

`related_files` может включать:

- исходный PDF;
- markdown-представление;
- промежуточный JSON;
- результаты OCR;
- таблицы извлечения;
- ручную разметку, если она есть.

## 8. Блок `extraction`

Блок фиксирует, как именно паспорт был получен.

Рекомендуемые поля:

- `pipeline_name`
- `pipeline_version`
- `extracted_at`
- `extractor`
- `parameters`
- `warnings`
- `errors`
- `coverage`

Назначение:

- обеспечить воспроизводимость;
- понимать ограничения паспорта;
- отделять проблему документа от проблемы извлечения.

## 9. Блок `document_structure`

Блок описывает агрегированную структуру раздела.

Рекомендуемые поля:

- `page_count`
- `page_roles`
- `sections`
- `toc`
- `body`
- `appendices`
- `page_layout_summary`

`page_roles` может содержать:

- титульные страницы;
- содержание;
- основное тело;
- графические листы;
- приложения;
- служебные страницы.

## 10. Блок `elements`

Блок хранит нормализованный перечень атомарных элементов.

Каждый элемент должен иметь минимальные поля:

- `element_id`
- `element_class`
- `page_no`
- `bbox`
- `source_ref`
- `confidence`
- `status`

Дополнительные поля:

- `text_raw`
- `text_normalized`
- `style_features`
- `graphic_features`
- `parent_element_id`
- `group_membership`
- `metadata`

Рекомендуемые значения `status`:

- `measured`
- `not_applicable`
- `missing`
- `low_confidence`

## 11. Блок `typical_element_groups`

Блок хранит устойчивые составные фрагменты из нескольких элементов.

Каждая типовая группа элементов должна иметь:

- `group_id`
- `group_class`
- `member_element_ids`
- `page_span`
- `bbox_union`
- `structure_signature`
- `text_signature`
- `graphic_signature`
- `repeat_count_in_section`
- `confidence`
- `status`

Назначение:

- сравнение на уровне устойчивых проектных фрагментов;
- выявление повторов;
- подготовка признаков для векторного сравнения.

## 12. Блок `text_axis`

Блок описывает текстовую ось анализа.

Рекомендуемая структура:

```text
text_axis
  text_inventory
  spectral_text_features
  semantic_text_features
  text_signatures
  text_quality
```

### 12.1. `text_inventory`

Состав текстовых данных:

- количество текстовых блоков;
- количество заголовков;
- количество пунктов;
- количество слов;
- количество символов;
- распределение текста по страницам.

### 12.2. `spectral_text_features`

Спектральные признаки текстового слоя:

- частоты символов;
- частоты слов;
- n-граммы;
- частоты типовых оборотов;
- длины абзацев;
- распределение терминов.

### 12.3. `semantic_text_features`

Семантические признаки:

- тематические признаки;
- семантические сигнатуры текстовых фрагментов;
- embedding-представления при наличии;
- признаки перефразирования;
- признаки смыслового совпадения.

### 12.4. `text_quality`

Качество текстового слоя:

- полнота извлечения;
- наличие OCR;
- доля нечитаемых фрагментов;
- признаки схлопывания таблиц в текстовый blob;
- предупреждения по кодировке.

## 13. Блок `graphics_axis`

Блок описывает графическую ось анализа.

Рекомендуемые поля:

- `graphics_inventory`
- `image_features`
- `vector_features`
- `layout_features`
- `graphic_signatures`
- `graphics_quality`

`graphics_inventory` может учитывать:

- количество изображений;
- количество векторных фрагментов;
- количество графических страниц;
- количество схем;
- количество предполагаемых узлов.

## 14. Блок `text_graphics_links`

Блок описывает связи текста и графики.

Рекомендуемые сущности:

- подписи к узлам;
- выноски;
- обозначения;
- текст внутри схем;
- привязки текстовых блоков к графическим объектам.

Каждая связь может иметь:

- `link_id`
- `link_class`
- `text_element_id`
- `graphic_element_id`
- `relation_type`
- `confidence`
- `source_ref`

## 15. Блок `feature_vectors`

Блок хранит векторные представления раздела.

Рекомендуемая структура:

```text
feature_vectors
  full_vector
  structural_vector
  text_spectral_vector
  text_semantic_vector
  graphics_vector
  text_graphics_link_vector
  typical_groups_vector
  vector_metadata
```

Для каждого вектора желательно хранить:

- `vector_id`
- `feature_space_id`
- `dimensions`
- `values`
- `availability_mask`
- `confidence_mask`
- `normalization`
- `created_at`

## 16. Блок `quality`

Блок описывает качество паспорта в целом.

Рекомендуемые поля:

- `overall_confidence`
- `extraction_coverage`
- `text_coverage`
- `graphics_coverage`
- `structure_coverage`
- `known_limitations`
- `excluded_features`

`excluded_features` должен фиксировать причину исключения признаков:

- `excluded_zero_variance`
- `excluded_not_applicable`
- `excluded_missing`
- `excluded_low_confidence`

## 17. Блок `lineage`

Блок описывает происхождение и связи паспорта.

Рекомендуемые поля:

- `previous_version_passport_id`
- `next_version_passport_id`
- `source_project_id`
- `derived_from`
- `notes`

Назначение:

- поддержка сравнения версий;
- анализ происхождения;
- будущий переход к уровню проекта целиком.

## 18. Блок `extensions`

Блок для будущих расширений.

Назначение:

- не ломать схему при появлении новых признаков;
- хранить экспериментальные данные;
- поддерживать развитие проекта без миграции каждого паспорта.

Требование:

- расширения не должны заменять обязательные блоки модели.

## 19. Минимальный состав MVP

Для первой версии паспорта обязательными следует считать:

- `schema`
- `identity`
- `source`
- `extraction`
- `document_structure`
- `elements`
- `typical_element_groups`
- `text_axis`
- `graphics_axis`
- `feature_vectors`
- `quality`

Можно отложить или оставить частично заполненными:

- `text_graphics_links`
- `lineage`
- `extensions`

## 20. Открытые вопросы

- Какой уровень детализации `bbox` нужен для первой версии.
- Нужно ли хранить сырые значения векторов в паспорте или вынести их в отдельный артефакт.
- Какой формат `availability_mask` и `confidence_mask` выбрать.
- Как именно версионировать `feature_space_id`.
- Какие признаки считать обязательными для минимального структурного вектора.
- Как отделить стабильные признаки от экспериментальных.
- Как хранить embedding-представления, если они появятся.

## 21. Следующий шаг

Следующий документ должен описать логическую модель `результата сравнения`.

Он должен быть спроектирован так, чтобы ссылаться на `section_passport`, не дублировать его содержимое и брать на себя всё, что зависит от состава библиотеки.
